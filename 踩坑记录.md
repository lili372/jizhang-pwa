# 踩坑记录

把项目中遇到的非直觉问题沉淀下来,避免重复踩坑,也方便以后回看决策依据。

---

## 1. Windows 下 `python -m http.server` 导致 Service Worker 注册失败

### 现象

浏览器控制台报错:

```
SW 注册失败 DOMException: Failed to register a ServiceWorker for scope (...)
with script (.../sw.js): The script has an unsupported MIME type ('text/plain').
```

电脑本地 `http://localhost:8000` 也复现,和手机、局域网无关。

### 根因

Python 标准库的 `http.server` 在 Windows 上会**读系统注册表**里的 MIME type 设置(通过 `mimetypes.init()` → `read_windows_registry()`)。

某些软件(Visual Studio、旧版 IIS、一些编辑器)在安装时会把注册表里 `.js` 的 `Content Type` 改成 `text/plain`。

浏览器规范规定:**Service Worker 脚本必须以 JavaScript MIME 返回**(`application/javascript`、`text/javascript` 等)。MIME 是 `text/plain` 就直接拒绝注册,整个 PWA 离线能力失效。

### 解决

项目根目录的 `serve.py` 自定义了一个 HTTP handler,强制覆盖 `extensions_map`,不再走注册表:

```python
class FixedMIMEHandler(SimpleHTTPRequestHandler):
    extensions_map = {
        **SimpleHTTPRequestHandler.extensions_map,
        '.js':   'application/javascript',
        '.mjs':  'application/javascript',
        '.json': 'application/json',
        ...
    }
```

顺便给所有响应加 `Cache-Control: no-store`,避免开发期"改了代码但浏览器用旧缓存"。

### 以后避坑

- **Windows 下所有本地起服务的项目,都不要用 `python -m http.server`**,直接写个 5 行的 `serve.py` 固化 MIME
- 遇到"SW 注册失败 / MIME 不对"的报错,第一反应先看浏览器返回的 Content-Type 是不是 JavaScript
- 验证办法:浏览器开发者工具 Network 面板,点 `sw.js` 请求,看 Response Headers 里的 `Content-Type`

## 2. 局域网 IP 的 http 协议不能注册 Service Worker

### 现象

手机 Edge/Chrome 打开 `http://192.168.x.x:8000`,页面能显示,但:
- 浏览器菜单没有"安装应用 / 添加到手机"选项
- `edge://serviceworker-internals` 里看不到对应条目
- 离线时点桌面图标打不开

### 根因

浏览器规范(Secure Contexts)规定,Service Worker 只能在**安全上下文**里注册:

- `https://` 任意地址
- `http://localhost`(或 `127.0.0.1`)
- 其他 `http://` 地址 → **拒绝**

局域网 IP `http://192.168.x.x` 不属于任何一类,默认被当成"不安全上下文",SW 无法注册。

### 解决

浏览器 flag `Insecure origins treated as secure` 可以**把指定的 http 地址当 https 对待**。

路径:`edge://flags` / `chrome://flags` → 搜 `Insecure origins` → 填入 `http://192.168.x.x:8000`(精确到端口,末尾不带斜杠)→ Relaunch。

### 以后避坑

- **生产环境别走这条路**,要么上 https,要么 localhost。flag 只是本地开发/自用的临时方案
- flag 里填的 URL 必须和实际访问的 URL **完全一致**(协议、IP、端口、不含斜杠)
- 如果以后给爸妈 / 其他人用,不能指望他们改 flag,必须走公网 https 部署

## 3. PWA 数据按 origin 隔离,IP 变了等于"换了个 App"

### 现象

手机 App 装好后数据记了一堆,某天电脑 IP 从 `192.168.1.23` 变成了 `192.168.1.50`,重新部署后,从新 IP 装的 App **看到的是空数据**。

看起来"数据丢了"。

### 根因

浏览器的 localStorage、IndexedDB、Cache Storage 等所有客户端存储,都按 **origin**(协议 + 主机 + 端口)做隔离。

- 旧 App 的 origin:`http://192.168.1.23:8000`
- 新 App 的 origin:`http://192.168.1.50:8000`

两个 origin 在浏览器眼里是完全不同的站点,存储互不可见。**数据没丢,只是被锁在旧 origin 的分区里访问不到了**。

### 解决

**主动防御**:

- 路由器后台绑定静态 IP(MAC 地址 → 固定 IP),让电脑每次连 WiFi 都分到同一个 IP
- 端口固定不变(`serve.py` 默认 8000,别随便改)

**事后补救**:

- 把电脑 IP 改回旧值(改路由器绑定,或电脑网络设置里手动指定)
- 如果改不回,就从之前导出的 JSON 备份里导入数据

**兜底**:

- App 里提供"导出/导入 JSON"功能
- 每次更新代码前强制导出一次,养成习惯

### 以后避坑

- **任何基于 localStorage 的本地应用,第一版就要做"导出/导入"功能**,比任何其他高级特性都重要
- 部署时 origin 就是"身份证",一旦变了数据就"消失",和应用内的逻辑 bug 是两回事
- 用户看到"数据丢了"时,**先问 URL 变了没有**,再去查代码

## 4. 不是所有"添加到主屏幕"都是 PWA

### 现象

手机上"添加到主屏幕"之后,点图标打开发现**还是在浏览器里**,带着地址栏和工具栏,不是独立 App 形态。

### 根因

Android 浏览器有两种把网页放到桌面的方式,看起来很像但本质不同:

| 操作名 | 实质 | 效果 |
|--------|------|------|
| **添加到主屏幕** | 网页快捷方式 | 点开在浏览器里打开,有地址栏 |
| **安装应用 / 添加到手机** | 安装 PWA | 独立窗口,无浏览器 UI,像原生 App |

只有满足 PWA 可安装条件(有 manifest、SW 注册成功、**安全上下文**)时,浏览器才会显示"安装应用"选项。

Edge 叫 "**添加到手机**",Chrome 叫 "**安装应用**",叫法不一样容易混淆。

### 解决

- 菜单里仔细找,带"安装"语义的才是 PWA 安装
- 地址栏右侧如果有一个"安装"小图标,点它最快
- 装完后检查:点桌面图标打开,**没有地址栏**才说明是真 PWA

### 以后避坑

- 遇到用户说"点图标还是网页",不要直接以为是 PWA 没配好,先问他当初点的是哪个菜单项
- 文档里明确写清两个浏览器的叫法,Edge 用户容易找不到"安装应用"
